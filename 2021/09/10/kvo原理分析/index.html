KVO原理分析

Posted on 2021-09-10

什么是KVO

苹果官方文档是这么介绍KVO的：

Key-value observing is a mechanism that allows objects to be notified of changes to specified properties of other objects.

意思就是说KVO是一种机制，这种机制允许某一个对象观察其他对象的属性时，假如被观察对象的属性发生变更的时候，观察对象可以收到通知。

KVO的使用

基本使用

  1. 调用addObserver: forKeyPath: options: context:注册观察者，其中context参数可以用作区分不同的上下文，而不是通过判断keyPath。
  2. 实现KVO的回调方法：observeValueForKeyPath:ofObject:change:context:
  3. 调用removeObserver:forKeyPath:移除观察者，当我们不需要观察这个属性变化的时候，可以调用这个方法。需要注意的是，观察者被释放前一定要调用这个方法，否则会导致crash。所以这个方法的调用可以放在dealloc里面。

KVO的底层实现

上文介绍了KVO最简单的用法，那么本节我们就来探索一下KVO的底层实现。

KVO只会对setter方法起作用

isa-swizzling

官方文档说明

我们来看一下官方文档对于KVO实现细节的说明：

Automatic key-value observing is implemented using a technique called isa-swizzling.

The isa pointer, as the name suggests, points to the object’s class which maintains a dispatch table. This dispatch table essentially contains pointers to the methods the class implements, among other data.

When an observer is registered for an attribute of an object the isa pointer of the observed object is modified, pointing to an intermediate class rather than at the true class. As a result the value of the isa pointer does not necessarily reflect the actual class of the instance.

You should never rely on the isa pointer to determine class membership. Instead, you should use the class method to determine the class of an object instance.

大概的意思是当我们对某一个对象进行观察的时候，这个对象的isa变量就会指向一个中间类，正是因为这个中间类的存在，才使得回调能够触发。

我们可以看到，中间类重写了name属性的getter方法。这样我们大概就能清楚的知道，KVO是通过重写中间类的setter方法来和父类的setter方法进行区分以便进行KVO的回调。那么为什么class方法也要重写呢，这是为了在调用中间类对象的class方法时返回父类，以便对于外部代码来说是不存在这个中间类的。比如本例中调用[self.room class]方法依旧是返回Room类而不是NSKVONotifying_Room类。

当我们调用removeObserver:forKeyPath:方法的时候对象的时候self.room这个对象的isa不会再指向中间类，会指向原始的类，比如这里的Room类。同时这个中间类并不会销毁，而是会一直存在在内存中。这一部分大家可以自己验证一下。

总结

  • KVO的实现最重要的就是通过runtime的机制动态生成一个中间类，一旦注册了观察者，对象的isa就会指向这个中间类，中间类通过重写对应属性的setter方法来实现回调方法的触发，这也是为什么KVO机制对于变量赋值不起作用的原因。
  • 在调用removeObserver:forKeyPath:方法移除观察者之后，对象的isa会指回原来的类而不是中间类，但是中间类还会继续存在在内存中方便复用。
  • 可以阅读苹果的源码来详细阅读KVO的实现。
